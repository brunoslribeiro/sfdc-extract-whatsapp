DOCUMENTACAO COMPLETA DO PROJETO
SFDC WhatsApp Extractor

1. OBJETIVO

Este projeto extrai conversas do Salesforce, salva os payloads localmente,
gera formatos auxiliares para auditoria e BI, importa os dados para MongoDB
e disponibiliza uma interface web simples para consulta das conversas.

O fluxo principal hoje usa a Conversation Data API para baixar as entries de
cada conversa. A descoberta das conversas a baixar e a estrategia incremental
ficam no exportador.

4. FLUXO FUNCIONAL

4.1 Descoberta das conversas

No modo principal, o projeto consulta o objeto Conversation via SOQL:

SELECT Id, StartTime, EndTime, LastModifiedDate, ConversationIdentifier
FROM Conversation
WHERE LastModifiedDate = LAST_N_DAYS:<dias>

Essa query identifica quais conversas foram alteradas dentro da janela.

4.2 Fila incremental

Depois de descobrir as conversas, o exportador compara cada
ConversationIdentifier com o arquivo de estado consolidado:

- se a conversa nao existe no estado, entra na fila
- se LastModifiedDate atual > lastModifiedDate salvo no estado, entra na fila
- se LastModifiedDate atual <= lastModifiedDate salvo no estado, ela e ignorada

Isso evita baixar novamente conversas que ja foram processadas sem mudanca.

4.3 Download das entries

Para cada conversa na fila, o projeto chama:

GET https://api.salesforce.com/platform/engagement/v1.0/conversation-entries

Parametros principais:
- conversationIdentifier
- recordLimit
- startTimestamp
- endTimestamp
- pageToken, quando houver paginacao

Se a org retornar nextPageToken, o cliente consolida todas as paginas em um
payload unico antes de salvar.

4.4 Persistencia por conversa

Para cada conversa baixada com sucesso:
- grava o JSON bruto
- escreve no NDJSON, se habilitado
- escreve no CSV, se habilitado
- atualiza imediatamente o estado consolidado
- atualiza o stage da execucao

Essa ordem e importante porque permite retomada real em caso de crash.


5. MODO INCREMENTAL E RETOMADA

5.1 Estado consolidado

O arquivo principal de estado fica em:

output/state/latest_state.json

Ele e indexado por ConversationIdentifier.

Cada item do estado pode guardar:
- conversationIdentifier
- sourceId
- startTime ou createdDate
- endTime
- lastModifiedDate
- sourceObject
- lastDownloadedAt
- lastRunId
- messageCount
- outputFile

5.2 Stage por execucao

Cada execucao gera uma pasta propria:

output/state/run_YYYYmmdd_HHMMSS/

Arquivos do stage:
- seen_conversations.json
- skipped_conversations.json
- downloaded_conversations.json
- failed_conversations.json
- summary.json

5.3 Retomada apos falha

Hoje o projeto suporta retomada pratica.

Como funciona:
- o estado consolidado e regravado durante a execucao
- se o processo cair depois de algumas conversas concluidas, essas conversas ja
  ficam no latest_state.json
- na proxima execucao, o exportador pula essas conversas e continua nas
  pendentes

Importante:
- uma conversa so entra no estado depois que o JSON dela foi escrito e os
  artefatos auxiliares daquela conversa foram processados
- se a conversa falhar, ela nao e marcada como concluida
- se falhar, ela aparece de novo em execucoes futuras

5.4 Onde ver falhas

Falhas ficam registradas em:
- output/logs/run_YYYYmmdd_HHMMSS/errors.csv
- output/state/run_YYYYmmdd_HHMMSS/failed_conversations.json

- colunas UTC auxiliares sao geradas

- formatType
- templateName
- attachmentCount
- senderRole
- senderAppType
- senderSubject
- clientTimestamp
- clientTimestampUtc
- serverReceivedTimestamp
- serverReceivedTimestampUtc
- transcriptedTimestamp
- transcriptedTimestampUtc
- sortTimestamp
- sortTimestampUtc
- relatedRecordsJson
- recipientsJson
- rawEntryJson

7.3 Colunas UTC auxiliares

As colunas abaixo facilitam ingestao em banco e ferramentas de ETL:

9. VIEWER

9.1 Tecnologia

- Flask
- leitura direta no MongoDB

9.2 O que o viewer faz

- lista conversas em ordem decrescente de lastMessageTimestamp
- busca por conversationId
- busca por texto do ultimo evento
- busca por numero/chave da plataforma

- abre a conversa em formato de chat
- permite filtrar por entryType
- permite ocultar eventos de sistema
- mostra rawEntry formatado para auditoria

O cliente respeita limitacao de taxa por minuto.

13. OBSERVACOES IMPORTANTES

13.1 Sobre o canal

No fluxo principal conversation-data, o projeto nao aplica filtro real por
channel na descoberta via Conversation.

Por isso:
- o params.json nao deve sugerir filtro de canal nesse modo
- o campo channel so deve existir no fluxo legado

13.2 Sobre a primeira execucao

A primeira execucao em uma janela tende a baixar mais dados, porque o estado
ainda nao existe ou esta vazio.

13.3 Sobre mudanca de layout de saida

Como os JSONs e CSVs agora ficam por execucao:
- output/json/run_*/
- output/csv/run_*/

isso melhora rastreabilidade e auditoria.

13.4 Sobre importacao

Se quiser importar uma execucao especifica em vez da mais recente, informe
diretamente a pasta da execucao:

python mongo_import.py --dir output/json/run_YYYYmmdd_HHMMSS --mongo-uri "mongodb://localhost:27017" --mongo-db sfdc_whatsapp


14. LIMITACOES

- o filtro por canal ainda nao e aplicado no fluxo principal da nova API
- o projeto depende de permissao de acesso a Conversation e a Conversation Data API
- se o enriquecimento opcional for usado, tambem depende de acesso a MessagingSession
- a interface web e uma ferramenta de auditoria local, nao uma aplicacao pronta para producao


15. RESUMO EXECUTIVO

Hoje o projeto:
- descobre conversas por Conversation
- baixa entries pela Conversation Data API
- grava estado incremental por ConversationIdentifier
- retoma bem apos falhas de processo
- organiza saidas por execucao
- gera CSV pronto para ETL com colunas UTC auxiliares
- importa para MongoDB
- oferece viewer local para auditoria

Esse arquivo serve como documentacao operacional e tecnica completa do projeto.
